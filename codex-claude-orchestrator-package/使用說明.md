# Codex + Claude Code Router 協作方案使用說明

## 1. 這份套件的目的

這份套件提供一套可在現有 `Claude Code + router + Codex` Docker image 中使用的主從式協作流程：

- 主控：`Codex`
- 執行：`Claude Code + router`
- 目標：降低主 agent context 過大與 token overflow 的風險

核心設計如下：

1. `Codex` 先將大任務拆成小 job。
2. 每個 job 僅允許少量檔案。
3. `Claude Code + router` 僅執行單一小 job。
4. 若 job 是 edit 類任務，系統會檢查檔案是否真的被修改。
5. 若 job 是 Python 測試檔任務，系統可自動跑 `python3 -m pytest -q` 驗證。

## 2. 主要內容

### 2.1 核心腳本

- `scripts/orchestrate_codex_to_claude.sh`
  - 主控協調器
- `scripts/worker_claude_router.sh`
  - Claude Code worker 執行器
- `scripts/evaluate_orchestration.sh`
  - 規劃與 inspect 類 smoke test
- `scripts/evaluate_worker_edit.sh`
  - edit 類 smoke test
- `scripts/run_codex_guarded.sh`
  - 低 context guarded 模式

### 2.2 啟動與設定

- `Dockerfile`
- `docker-compose.yml`
- `router-config.json`
- `launch-docker-codex.ps1`
- `launch-docker-codex-guarded.ps1`
- `run-claude.sh`
- `run-codex.sh`

### 2.3 驗證用範例專案

- `hello-python/`

## 3. 已驗證的能力

本次已驗證下列行為：

1. 主 `Codex` 可先拆 job，再派給 `Claude Code + router`。
2. planner 輸出的檔案路徑會被校正為 scope 內正確相對路徑。
3. inspect 類任務可在小 scope 下完成，不需整包讀 repo。
4. edit 類任務若模型只口頭說成功、但沒有實際改檔，系統會判為失敗。
5. Python 測試檔類型的 edit 任務可自動執行 `python3 -m pytest -q` 驗證。

## 4. 使用方式

### 4.1 規劃 + inspect 類驗證

在 container 內執行：

```bash
cd /workspace/linux_remote/ubuntu22-claude-ccr
bash ./scripts/evaluate_orchestration.sh
```

### 4.2 worker edit 類驗證

在 container 內執行：

```bash
cd /workspace/linux_remote/ubuntu22-claude-ccr
bash ./scripts/evaluate_worker_edit.sh
```

### 4.3 實際派工

```bash
cd /workspace/linux_remote/ubuntu22-claude-ccr
bash ./scripts/orchestrate_codex_to_claude.sh "Edit tests/test_placeholder.py to replace assert True with assert 1 + 1 == 2" /workspace/linux_remote/ubuntu22-claude-ccr/hello-python
```

## 5. 重要判定規則

### 5.1 何時算 edit 成功

edit 類 job 需要同時滿足：

1. worker 回傳成功狀態
2. 目標檔案 hash 真的有變
3. 若有測試命令，測試必須通過

### 5.2 何時會被擋下

以下情況會被擋下，不算成功：

1. 模型說有改，但檔案完全沒變
2. 檔案有變，但驗證命令失敗
3. 路徑不正確，worker 找不到指定檔案

## 6. 已知限制

1. 本方案可以降低 token overflow 風險，但不能保證所有本地小模型都穩定完成複雜 agentic edit。
2. 小型 open-source model 仍可能出現：
   - 只描述修改、不真的動手
   - 判斷錯誤
   - 對較大任務品質不穩
3. 如果 worker 模型品質不足，建議升級較強模型，再沿用同一套 orchestration 架構。

## 7. 建議排查順序

如果流程不如預期，建議依序檢查：

1. Windows host 的 Ollama / OpenAI-compatible endpoint 是否正常
2. Docker container 是否能連到 `host.docker.internal`
3. `Claude Code router` 是否健康
4. `Codex` planner 是否成功輸出可解析 JSON
5. worker 的 `status.json` 中：
   - `actual_changed_count`
   - `verification_note`
   - `test_exit_code`
